# 章节写作参考

> 这是参考，不是强制模板。不同章节可以有不同的组织结构。

# 第 N 章：章节标题

## 问题切入

从一个具体场景开始：

- 业务背景是什么？
- 当前数据量、查询方式或协作方式遇到了什么问题？
- 如果继续用上一章的方法，会出现什么瓶颈？

不要从术语定义开始，不要写"随着大数据时代的发展..."这类抽象开场。

## 核心判断

用 1 到 3 段给出本章最重要的判断。每章的引题方式应该不同。

不要用固定句式。以下是几种可选的方式：
- 一个反常识的观察
- 一个具体的问题及其答案
- 一个对比（单机能做什么 → 为什么单机不够了）
- 一个判断句（"这一章要解决的问题是..."）

## 机制解释

解释内部机制：

- 数据如何组织？
- 查询或计算如何发生？
- 写入、读取、存储、延迟、一致性之间有什么取舍？
- 哪些条件会让机制失效？

## 系统位置

说明本章概念处在整条演化路径的哪个位置：

- 它继承了前一章的什么问题？
- 它把什么能力推进了一步？
- 它还没有解决什么问题？
- 它为什么会引出下一章？

## 场景案例（可选）

用一个接近真实平台的案例说明。只有当场景贯穿全章能显著帮助理解时才写。

## 常见误区（可选）

挑真正高频、真正有害的认知错误。不要凑数。

## 实战任务（可选）

当章节概念需要动手才能建立直觉时才设计。任务必须能验证本章的核心判断。

## 小结引出下一章

总结本章解决了什么问题，并明确指出下一个未解决的问题。
