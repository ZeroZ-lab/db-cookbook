# 书稿扩写与验收计划

本文档把“由浅入深、系统完整、细节充分”的目标拆成可检查的写作标准。后续扩写正文时，不以“章节存在”为完成标准，而以读者是否能从 PostgreSQL 基础能力逐步迁移到大数据和 AI 数据基础设施判断能力为完成标准。

## 一、最终交付标准

本书完成时应满足四个条件：

1. **路径清楚**：读者能看出知识如何从 PostgreSQL 基础、SQL 分析、大表边界、OLTP/OLAP 分化，逐步进入数仓、ETL/CDC、批处理、实时计算、OLAP、湖仓、向量、图和治理。
2. **机制充分**：每个重要概念都说明它解决什么问题、不解决什么问题、为什么出现、和前后系统的关系，以及在真实平台中如何落地。
3. **章节一致**：每章都能对应“问题切入、核心判断、机制解释、系统位置、场景案例、常见误区、实战任务、小结引出下一章”八段结构。
4. **可操作**：每章至少有一个实战任务，任务包含数据背景、操作步骤、观察指标、对比实验和复盘问题。

## 二、读者递进路径

### 阶段 1：从会查数据到理解数据库

对应章节：第 0-3 章。

目标能力：

- 理解数据库不是表格集合，而是业务事实、约束、事务、索引、执行计划和存储组织的组合。
- 能把一个业务对象拆成表、字段、主键、外键、约束和事务边界。
- 能写基础分析 SQL、聚合 SQL、多表 JOIN、窗口函数和常见指标 SQL。
- 能用分区、索引、物化视图和执行计划理解 PostgreSQL 大表边界。

验收问题：

- 读者能否解释为什么 `SELECT *`、无口径聚合、错误 JOIN 粒度会造成分析风险？
- 读者能否说明索引、分区、物化视图分别解决什么问题，又不能解决什么问题？
- 读者能否看懂一段 `EXPLAIN ANALYZE` 的主要节点，并判断慢查询瓶颈来自扫描、连接、排序还是聚合？

### 阶段 2：从单机数据库到数据平台

对应章节：第 4-9 章。

目标能力：

- 理解 OLTP 和 OLAP 的分化不是工具偏好，而是读写模式、数据规模、延迟目标和一致性要求不同。
- 能把 PostgreSQL 业务表迁移成数仓事实表、维度表、指标层和数据集市。
- 理解 ETL、ELT、CDC、批处理、实时计算和 OLAP 数据库在链路中的位置。
- 能区分 Hive、Spark、Trino、Kafka、Flink、ClickHouse、Doris、DuckDB 的系统角色。

验收问题：

- 读者能否说明为什么业务库不能承担所有历史分析？
- 读者能否设计一条从 PostgreSQL 到数仓、OLAP 和实时指标的基础链路？
- 读者能否判断一个场景应该用批处理、实时流、交互式查询还是 OLAP 数据库？

### 阶段 3：从数据平台到智能数据系统

对应章节：第 10-13 章。

目标能力：

- 理解向量数据库解决语义相似度检索，不等于解决事实正确性、权限控制、结构化分析和数据治理。
- 理解图数据库解决多跳关系、路径和网络结构问题，不等于替代关系数据库或数仓。
- 理解湖仓架构解决低成本存储、开放格式、统一管理和多引擎访问问题，但仍需要建模、治理和质量控制。
- 理解数据治理、元数据、血缘、质量、权限和指标口径是 AI 数据系统可信运行的基础。

验收问题：

- 读者能否解释 RAG 链路中原始文档、分块、向量、元数据、权限和召回结果之间的关系？
- 读者能否说明知识图谱和 GraphRAG 解决的是哪类关系问题？
- 读者能否描述一个数据资产从源系统进入湖仓、被建模、被治理、被 AI 应用使用的完整生命周期？

### 阶段 4：从学习知识到构建系统

对应章节：第 14-17 章。

目标能力：

- 能把 PostgreSQL、SQL、数仓、批流、OLAP、湖仓、向量、图和治理组合成项目。
- 能按学习顺序判断当前短板，不再把工具名当成能力本身。
- 能用能力地图检查自己是否具备建模、链路、选型、治理和 AI 数据系统理解能力。

验收问题：

- 读者是否能独立设计一个“业务库 -> 数据平台 -> AI 数据应用”的最小闭环？
- 读者是否能说明每个组件加入系统的原因、代价和边界？
- 读者是否能为项目写出可验证的指标、任务、质量检查和复盘问题？

## 三、单章扩写标准

每章扩写时必须完成以下检查：

| 检查项 | 合格标准 |
| --- | --- |
| 问题切入 | 以真实业务或平台问题开头，不从术语定义开头 |
| 核心判断 | 明确说明本章机制为什么出现、解决什么、不解决什么 |
| 机制解释 | 讲清数据组织、读写路径、计算方式、代价和失效条件 |
| 系统位置 | 说明它继承上一章什么问题，又引出下一章什么问题 |
| 场景案例 | 使用电商、内容、订单、行为事件、知识库或指标平台等连续场景 |
| 常见误区 | 至少 3 个误区，每个误区说明错因和正确边界 |
| 实战任务 | 包含数据准备、操作步骤、观察指标、对比实验、复盘问题 |
| 承上启下 | 结尾必须指出当前机制未解决的问题 |

## 四、内容深度分层

为了保证“从浅到深”，每个核心概念按三层写：

1. **直觉层**：先用业务场景说明它为什么需要出现。
2. **机制层**：解释内部结构、计算路径、读写代价和约束。
3. **工程层**：说明它在真实平台中的落地方式、常见组件、质量指标和故障模式。

例如写索引时，不只写“索引提高查询速度”，而要写：

- 直觉层：表变大后，全表扫描会让点查变慢。
- 机制层：索引用额外存储和写入维护成本换取特定访问路径上的更少读取。
- 工程层：业务库用 B-tree 支持订单点查，日志表用 BRIN 辅助时间范围扫描，搜索或 JSON 场景可能使用 GIN，但索引不能替代数仓建模和 OLAP 引擎。
