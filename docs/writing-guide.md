# 写作规范

## 一句话主张

这份内容要让已经会写基础 SQL 的读者理解：数据库不是一组孤立技术点，而是一条从业务数据管理、分析计算、平台工程到 AI 数据基础设施持续演化的系统路径。

## 读者定位

主要读者：

- 会写基础 SQL，能理解表、字段、过滤、聚合、连接。
- 用过 PostgreSQL、MySQL、数据分析工具或业务报表，但对底层机制不系统。
- 想从“会查数据”提升到“能理解并参与构建数据平台”。

不默认读者已经掌握：

- 分布式系统。
- 数仓建模。
- 流计算。
- 向量检索。
- 图计算。
- 数据治理体系。

## 写作总路线

采用演化式叙事，而不是概念枚举。

章节可以覆盖很多工具名，但每个工具名都必须服务于系统位置判断。写作时先说明它为什么出现、解决哪类系统压力，再说明它和 PostgreSQL、数仓、湖仓、向量检索、图关系分析或治理体系的连接方式。

每一部分都要回答一组递进问题：

1. 单个业务系统如何可靠地组织和修改数据？
2. 当查询从业务操作变成分析问题时，数据库会遇到什么压力？
3. 为什么一个数据库很难同时服务高频交易和复杂分析？
4. 为什么数据需要被抽取、同步、建模、分层和治理？
5. 为什么 AI 应用需要新的数据组织方式和可信数据底座？

## 大纲层级规则

主大纲采用“阶段 -> 核心判断 -> 机制模块 -> 阶段成果”的结构。

- 阶段用于表达学习迁移路径。
- 核心判断用于防止章节变成工具清单。
- 机制模块用于展开必要概念。
- 阶段成果用于定义读者完成本阶段后应具备的能力。
- 项目实战用于把 PostgreSQL、SQL、数仓、批流、OLAP、湖仓、向量、图和治理串成闭环。

## 已验证写作样式

从既有章节草稿中，保留以下写作样式作为后续默认标准：

- 每节标题使用“概念 + 机制判断”的形式，例如“PostgreSQL 核心结构：数据系统是如何被组织起来的”“多表关联：从分散数据中恢复完整业务事实”。
- 开头先建立读者误区或真实场景，再给核心判断，不直接进入定义。
- 正文用电商数据模型贯穿：`users`、`products`、`orders`、`order_items`、`payments`、`events`。
- 每个概念都要落到“它在业务系统里代表什么”，例如 Table 表达业务对象、事件或关系，Row 表达一条事实，Column 描述事实属性。
- 小节之间要有能力递进：基础查询解决取数边界，聚合分析把明细变成指标，JOIN 从分散表中恢复完整业务事实。
- 章节小结要回到系统演化：PostgreSQL 业务库 -> ETL / CDC -> 数仓 / OLAP / 湖仓 / 实时计算 -> BI / 指标 / AI 应用。

## 配图风格

章节配图采用“信息优先、结构优先、阅读优先”的编辑式知识图解风格。

优先使用：

- 层级图：表达 PostgreSQL Server、Database、Schema、Table、Row / Column 的组织关系。
- 机制图：表达明细记录如何经过过滤、聚合、分组变成指标。
- 关系图：表达 `users -> orders -> order_items -> products` 这类业务对象如何通过键连接。
- 对比图：表达 `INNER JOIN`、`LEFT JOIN`、`FULL JOIN`、`CROSS JOIN`、`SELF JOIN` 的差异。

避免把配图做成装饰海报。图要帮助读者理解结构、关系、路径和边界。

## 每章固定结构

每章都按以下顺序写：

1. **问题切入**：从真实场景开始，不从术语定义开始。
2. **核心判断**：先给读者一个可以带走的判断。
3. **机制解释**：解释内部机制、关键约束、代价和边界。
4. **系统位置**：说明它在 PostgreSQL、数仓、数据平台或 AI 基础设施中的位置。
5. **场景案例**：用接近真实平台的案例说明它如何落地。
6. **常见误区**：指出概念滥用、过度简化或错误迁移。
7. **实战任务**：给读者一个能验证理解的操作任务。
8. **小结引出下一章**：总结本章解决了什么问题，并自然引出下一个系统问题。

## 概念处理规则

每个重要概念都必须回答五个问题：

- 它解决什么问题？
- 它不解决什么问题？
- 它为什么出现？
- 它和前后系统是什么关系？
- 它在真实数据平台中如何落地？

不要写成：

> 索引是一种用于提高数据库查询效率的数据结构。

应该写成：

> 当表从几千行增长到几千万行时，全表扫描会把一次点查变成一次大规模数据读取。索引的作用不是让所有查询都变快，而是用额外的存储和写入维护成本，换取特定访问路径上的更少读取。它是单机数据库继续支撑业务查询的关键机制，但无法解决跨系统分析、冷热数据分层或全局数据建模问题。

## 风格要求

保持：

- 专业。
- 系统。
- 清晰。
- 机制导向。
- 重视边界和代价。
- 重视迁移关系。

避免：

- 词典式堆砌。
- 工具清单式介绍。
- 只讲优点不讲边界。
- 用大词替代机制。
- 把 AI 数据系统写成向量数据库单点能力。
- 把大数据平台写成 Spark、Flink、ClickHouse 等工具名集合。

## 对比写法

需要大量使用结构化对比，例如：

| 问题 | PostgreSQL 内部解法 | 数据平台阶段的解法 | AI 数据系统阶段的变化 |
| --- | --- | --- | --- |
| 快速查找 | 索引 | OLAP 引擎、预聚合、列式存储 | 向量索引、混合检索 |
| 历史分析 | SQL 聚合、物化视图 | 数仓分层、批处理 | 特征、上下文、评测数据集 |
| 一致性 | 事务、约束、锁、MVCC | CDC、数据质量、血缘 | 可追溯上下文、权限和治理 |

## 实战任务标准

实战任务不应只是“写一条 SQL”。它应该让读者验证本章判断。

一个合格任务通常包含：

- 数据背景。
- 要解决的问题。
- 初始做法。
- 遇到的瓶颈。
- 机制性改进。
- 观察指标。
- 复盘问题。

示例：

> 给一张订单表插入 100 万行模拟数据。先写一个按用户和时间范围查询订单的 SQL，观察执行计划。再添加组合索引，对比扫描行数、执行时间和写入成本。最后回答：这个索引解决了什么查询问题？它是否能提升所有订单分析？
