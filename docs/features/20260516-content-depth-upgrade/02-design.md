 ## 设计目标

 1. 每章升级后的内容让读者感受到**三层深度**（直觉→机制→工程），而非两层
 2. 机制层不追问"为什么→怎么"→代价"， 范围"， 而 工程层让读者获得**判断力**（选型对比表、故障诊断、 处理策略）
 3. 纵向主线让读者在每章末尾看到"整个系统如何演进"

 而 场景连续性让读者看到**同一数据**如何从 PG 到数仓再到湖仓和 RAG 直到治理的完整路径

## 关键设计决策
### KD1: 章节升级模板

每个 section-complete 文件的升级方式按 **KD5 质量分级** 决定：
- A 级（P1-P6 ≥ 4/6）：增量修改（补缺失模式段落）
- B 级（P1-P6 ≥ 2/6）：重点补写（补核心判断、边界确认、演进链接段落）
- C 级（P1-P6 ≤ 1/6，YAML模板/功能清单）：全量重写为论证式格式

每个升级后的 section 必须满足：
- **开头**：用问题或场景引入，不用 YAML 对话或功能清单开头
- **机制路径**：按"问题→计算路径→代价/失效条件"展开
- **边界句**：明确"解决什么/不解决什么"
- **演进链接**：1-2句向前连接到后续系统

新增内容写入 main file 的以下位置：
- **场景案例段**：对比表（含推荐场景行+注意事项列）
- **常见误区段**：故障清单（含分类/症状/检测/缓解/严重度）
- **小结引出下一章段**：T1-T5 五步过渡 + 5条主线回溯+推进桥段

场景连续性修复：增量修改 main file 的问题切入、核心判断、场景案例段，使其引用本章指定电商子场景（spec 4.4）

### KD2: 章节过渡模式（T1-T5 五步）+ 纵向主线桥段

每章 main file "小结引出下一章"段遵循 Ch 1 的五步过渡模式（line 1538-1601）：

**T1 反总结**：显式说明本章未完成什么，如果只是浅层学习会怎样（1-2句）
**T2 核心判断重述**：升格为原则级洞察（blockquote格式，1句）
**T3 直觉枚举**：2-4个"不是X而是Y"对比（每对比1句+可选箭头链图）
**T4 迁移声明**：前后能力框架："从'XX'到'YY'的迁移"（1句）
**T5 桥段**：下一章主题+问题化展开（2-3句）

在 T5 之后嵌入纵向主线桥段（每主线2-3句：回溯+推进+未解之问），只覆盖与该章最相关的2-3条主线。

完整模板示例（Ch 5→6）：

```markdown
## 小结引出下一章

到这里，第五章完成的不是一次数仓建模入门。如果只是把维度建模当成"画星形图"，我们很容易陷入表结构设计而忽略建模真正解决的问题——让数据在规模增长后仍然有口径、有层次、有质量边界。

> 数仓建模的价值不只是画出事实表和维度表，而是把业务数据组织成有口径、有层次、可下钻、可复用的指标体系。

- 数据组织不是随便建表，而是按业务过程和维度建模
- 分层不是仓库货架，而是口径递进和质量隔离
- 指标不是 SQL 聚合结果，而是有定义、有口径、有归属的分析对象

到这里，我们已经完成了从"数据存在数据库里"到"数据被建模成可分析体系"的迁移。

下一章要回答的问题是：建模解决的是数据如何被组织的问题，但数据如何从业务库到达数仓——抽取、同步、一致性保证——建模本身无法回答。ETL 和 CDC 就是让数据在跨系统间流动并保持一致的机制。

---

**主线递进**：

> **一致性线**：Ch 5 用数仓建模约束了数据组织的一致性，但建模不解决跨系统同步的一致性——数据从 PG 到数仓如何保证不丢不重？这是 Ch 6 的 CDC 和 ETL 要回答的。
> **建模线**：Ch 5 把 orders 建模成事实表+维度表。但建模后的数据如何被批量处理、流处理和 OLAP 引擎消费？建模线在 Ch 7-9 会展示建模成果如何被不同计算引擎使用。
> **故障与边界线**：Ch 5 的建模退化（分层坍缩、口径冲突）是数仓最常见的故障。Ch 6 的 ETL 失败会加剧这些退化——上游断点导致下游指标静默出错。
```
  每章 main file 末尾增加系统位置桥段 + 场景位置桥段，格式：
   - **系统位置**： "本章解决了 [X 问题]， → 下一步要解决 [Y 问题]"
   - **场景位置**： "[orders 数据在这里阶段处于 Z 位置] → 下一步进入 W 位置"

 | 章节 | 系统位置桥段 | 场景位置桥段 |
 |------|-------------|---------------|
 | Ch 2→3 | SQL 是数据库查询能力的上限 | orders 分析遇到大表变慢 |
 | Ch 3→4 | 单机边界迫使 OLTP/OLAP 分化 | orders 在 PG 做交易变慢 |
 | Ch 4→5 | OLAP 需要数仓建模 | orders 进入数仓需要建模 |
 | ... | ... | ... |

 场景位置桥段与系统位置桥段**分离**——这是 Eng Scout 出的关键修正。

### KD3: 工程层格式模板设计
 #### 对比表模板
 固定结构：

 | 维度 | 选项 A | 选项 B | 选项 C | 注意事项 |
 |------|--------|--------|--------|----------|
 | [维度 1: 具体值] | [维度 1: 具体值] | [维度 1: 具体值] | [注意事项文本] |
 | ... | ... | ... | ... |
 | **推荐场景** | 什么场景选 A | 什么场景选 B | 什么场景选 C | -- |

 规则：
 - 每行维度必须包含**具体数值或场景触发条件**，不能只写功能标签
 - "推荐场景"行必须给出明确的选择指引
 - "注意事项"列必须包含**注意事项或边界条件**

 #### 故障清单模板
 固定结构：

 | 分类 | 故障模式 | 具体症状 | 检测方法 | 缓解措施 | 严重度 |
 |------|----------|----------|----------|----------|--------|
 | [分类] | [具体故障名] | 必须写**具体症状** | [检测方法] | 缓解措施] | 高/中/低 |

 规则：
 - 每项故障必须包含：**具体症状** + **根因引用本章机制** + **缓解措施不是"用 X 工具"**
- 严重度用高/中/低，不量化

 示例（对比表）：

 | 维度 | PostgreSQL OLTP | ClickHouse OLAP | 注意事项 |
 |------|----------------|----------------|--------------------|
 | 点查延迟 | < 1ms (索引命中) | ~50ms (稀疏索引) | PG 瓏索引但 B-tree；CH 依赖 MergeTree 範 |
 | 聚合范围 | 单表/小范围 | 大表/全表 | PG 用 CTE 分步聚合；CH 铉并行计算 |
 | 写入吞吐 | 适合高频小写 | 适合批量导入 | PG 适合事务写入；CH 异步写入 |
 | **推荐场景** | 业务查询+事务保障 | 大规模分析+报表 | PG 优先保障事务完整性；CH 优先保障分析吞吐量 |

 示例（故障清单）：

 | 分类 | 故障模式 | 具体症状 | 检测方法 | 缓解措施 | 严重度 |
 |------|----------|----------|----------|----------|--------|
 | 数据一致 | 业务库与数仓 GMV 不一致 | 比较两系统同日 GMV | 统一对账任务+口径 SLA | 高 |
 | 索引退化 | 知识库文档更新后索引未同步 | 搜索返回过期内容 | 定期重建索引+版本追踪 | 中 |
 | 分区不均 | 知识库某分区数据量 40x 其他分区 | ANN 检索延迟不均匀 | 预分区+数据 salting | 中 |

 ### KD4: 纵向主线回溯+推进桥段设计
 每章 main file "小结引出下一章"段末尾添加 2-3 句主线桥段，格式：

 > **主线回溯**： [上一章在本主线解决了什么问题]。
 > **主线推进**： [本章在本主线打开了什么新问题]。
 > **未解之问**： [下一章在本主线要解决什么]？

 5 条主线不独立成附录，而是嵌入每章末尾。2-3 句桥段。
  每章必须覆盖至少 2 条主线（从 5 条中选择与该章最相关的 2-3 条）。

 主题追踪矩阵确保每章每主线新增维度不重复前一章。

### KD5: Section-complete 机制层质量升级标准
 识别公式化浅层内容的具体特征并定义升级标准：

 | 特征 | 公式化/浅层 | 深度升级后 |
 |------|----------------|----------------------|
 | 开头方式 | yaml 对话开头 (`资深工程师: ...`) | 问题引入 (`当 orders 表从万行增长到千万行...`) |
 | 概念展开 | 功能清单枚举 (`支持 X、Y、Z`) | 问题→路径→边界 (`为什么 X 出现→ X 如何工作→ X 代价和失效条件`) |
 | 概念间连接 | 独立条目无关联 | 每个概念说明与前/后系统的关系 |
 | 电商场景 | 不引用或引用一次后遗忘 | 在 ≥ 3/10 小节中引用电商场景 |
 | 边界句 | 无"不解决什么"表述 | 明确"解决 X / 不解决 Y" |

 升级方式：**增量修改**，不是全量重写。
 - 公式化开头 → 重写为问题引入
1-2 句)
 - 功能清单 → 重写为问题→路径→边界三段式（每概念 50-150 字）
 - 无关联 → 添加 1-2 句前后系统关系
 - 场景缺失 → 添加 1-2 句场景引用
 - 边界句缺失 → 添加 1 句边界句
