# 质量审校清单详细标准

## Q1: 问题切入是否使用本章指定连续场景？

| 维度 | 标准 |
|------|------|
| yes 条件 | 本章指定场景关键词（如 orders/GMV/数仓/RAG/湖仓/治理）在"问题切入"段出现 ≥ 3 次 |
| regex 检测 | 在 main file 的 `## 问题切入` 段中搜索场景关键词 |
| 人工要点 | 检查场景引用是否具体（如"orders 表 GMV 指标口径不一致"而非泛化的"数据问题"） |

## Q2: 核心判断是否明确"解决什么/不解决什么"？

| 维度 | 标准 |
|------|------|
| yes 条件 | 有显式"不解决"或"不等于"或"不能替代"表述 |
| regex 检测 | `不解决|不等于|不能替代|不是.*的全部|边界` |
| 人工要点 | 确认"不解决"表述针对的是真实常见误解，不是虚构的 |

## Q3: 场景案例是否引用本章指定电商/数仓/RAG 子场景？

| 维度 | 标准 |
|------|------|
| yes 条件 | "场景案例"段包含本章指定子场景的具体数据/表名/查询/流程 |
| regex 检测 | 在 `## 场景案例` 段搜索子场景关键词 |
| 人工要点 | 确认子场景引用是连续的（和 Ch 1-9 的 orders/数仓场景衔接） |

## Q4: 机制层是否按"问题→路径→边界"展开，而非功能清单枚举？

| 维度 | 标准 |
|------|------|
| yes 条件 | section-complete 文件开头不是 YAML 对话（`资深工程师:`/`数据分析师:` 等角色标签）；不是纯功能列表段（`特点/优势/劣势` 模板）；有"为什么出现"的提问或张力 |
| regex 检测 | 在 section 文件中搜索 `资深工程师|数据分析师|数据工程师|新同事|特点|优势|劣势|适用场景` YAML标记 |
| 人工要点 | 确认每个概念有"问题→机制路径→代价/边界"三段式结构 |

## Q5: 每个核心概念是否说明代价和失效条件？

| 维度 | 标准 |
|------|------|
| yes 条件 | 有代价表述（如"额外存储"、"写入维护成本"、"性能代价"）或有失效条件表述（如"不能解决"、"失效条件"、"当...时不适用"） |
| regex 检测 | verify-writing-contract 已覆盖（搜索 `代价|失效|不解决|不能|边界|限制|前提|前提条件`） |
| 人工要点 | 确认代价/失效条件是具体的（如"索引不能解决跨系统分析"而非泛化的"有局限性"） |

## Q6: 机制层是否使用连续场景说明计算路径？

| 维度 | 标准 |
|------|------|
| yes 条件 | 本章指定场景在 ≥ 3/10 个 section 中被引用（不是只引1次就遗忘） |
| regex 检测 | 在 section 文件中搜索场景关键词，计数 ≥ 3 |
| 人工要点 | 确认场景是"累积"的（后续section有比前更深的场景细节），而非"提及" |

## Q7: 工程层对比表是否有"推荐场景"行和"注意事项"列？

| 维度 | 标准 |
|------|------|
| yes 条件 | Markdown 表格有 `**推荐场景**` 行（加粗标记）和 `注意事项` 或 `Notes` 或 `Caveats` 列 |
| regex 检测 | `\*\*推荐场景\*\*` 和 `注意事项` 在 main file `## 场景案例` 或 `## 常见误区` 段出现 |
| 人工要点 | 确认表格格式正确（列头含注意事项列、行含推荐场景行） |

## Q8: 工程层对比表单元格是否包含具体场景触发条件？

| 维度 | 标准 |
|------|------|
| yes 条件 | 至少 1 个单元格包含具体数值或场景描述（如"点查订单详情，延迟<10ms"），不能只写功能标签（如"低延迟"） |
| regex 检测 | **纯人工**——regex无法判断"具体vs泛化" |
| 人工要点 | 逐单元格检查：是否有数字、具体场景名、具体延迟/吞吐量指标 |

## Q9: 故障清单每项是否有具体症状+根因+缓解措施？

| 维度 | 标准 |
|------|------|
| yes 条件 | 至少 3 项故障满足三要素：①具体症状（有具体数据/场景名）②根因引用本章机制③缓解措施是具体操作（不是"用X工具"） |
| regex 检测 | **纯人工**——regex无法判断三要素完整性 |
| 人工要点 | 每项故障检查：症状是否具体？根因是否引用本章机制？缓解是否可操作？ |

## Q10: "小结引出下一章"是否有主线回溯+推进桥段？

| 维度 | 标准 |
|------|------|
| yes 条件 | main file `## 小结引出下一章` 段末尾有 ≥ 2 条主线的回溯+推进+未解之问桥段 |
| regex 检测 | 搜索 `主线回溯|主线推进|数据组织线|检索线|一致性线|建模线|故障与边界线` |
| 人工要点 | 确认桥段格式正确（回溯+推进+未解之问），内容不重复前一章 |
