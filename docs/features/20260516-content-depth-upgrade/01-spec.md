## 1. 上下文与动机

db-cookbook 是一套 18 章、5 部分的系统化课程，从 PostgreSQL 到 AI 数据基础设施。

### 当前内容组装方式

内容由两层文件组装：
- **Main file**（~7-10k chars）：8-segment 骨架（问题切入、核心判断、机制解释头、系统位置、场景案例、常见误区、实战任务、小结引出下一章）
- **Section-complete files**（10-12 个/章，每个 ~3-8k chars）：机制解释的详细展开
- `generate-site.mjs` 将 section-complete 内容插入 main file 的"机制解释"段与"系统位置"段之间

### 核心差距

1. **机制层质量不均**：Ch 7 s1/s5、Ch 9 s1 采用公式化对话开头、功能清单式枚举而非问题→机制→边界路径。不能假设所有 section-complete 内容只需补工程层。
2. **工程层几乎缺失**：所有章节均缺少真实平台落地方式、故障模式、选型判断。
3. **场景连续性断裂**：电商模型只在 Ch 1-2 使用，后续章节换了场景。
4. **跨章演进视角缺失**：读者只看到单章深度，看不到整条链路如何演进。

## 2. 目标

将 Ch 2-13 每章升级到 Ch 1 水平：机制充实（问题→路径→边界而非功能清单）、实战任务有闭环、边界句明确、对比表+故障清单表达工程层。不追求出版最终定稿。

嵌入 5 条纵向主线为"回溯+推进"桥段，让读者在每章末尾看到跨链路演进视角。

## 3. Scope

### 升级章节

Ch 2-13（共 12 章）。Ch 0 定位章和 Ch 1 标杆章保持不动。Ch 14-17 收束章节后续单独处理。

### 升级目标文件

- **Section-complete 文件**：机制层质量审校和补写（主要工作量）
- **Main file**：工程层增量添加 + 场景连续性修复 + 纵向主线桥段
- **不全量重写 main file**，保持 8-segment 骨架不动

### 纵向主线

5 条贯穿 Ch 0-13 的主题线，**嵌入每章"小结引出下一章"段**作为 2-3 句"回溯+推进"桥段，不独立成附录。

| 主线 | 覆盖章节 | 起点概念 | 终点概念 |
|------|----------|----------|----------|
| 数据组织线 | 1→3→4→5→7→9→12→10→11→13 | 表/行/列 | 向量/图/治理对象 |
| 检索线 | 1→3→4→9→10→11→13 | 索引/B-tree | ANN/图遍历/血缘发现 |
| 一致性线 | 1→3→6→8→12→10→11→13 | 事务/MVCC | CDC/Exactly Once/快照隔离/向量版本/治理审计 |
| 建模线 | 1→2→5→7→9→12→10→11→13 | ER/表 | 星/雪花/宽表→湖仓建模→分块+Embedding→本体→指标口径 |
| 故障与边界线 | 3→4→6→7→8→9→10→11→12→13 | 大表瓶颈 | OLTP 压力→ETL 失败→Shuffle→Watermark 延迟→召回噪声→图幻觉→治理失效 |

### 不在 Scope 内

- Ch 0 定位章、Ch 1 标杆章重写
- Ch 14-17 收束章节深度升级
- 出版级编辑（术语全文统一、事实核查闭环）
- 交付形式改造（交互式 SQL Lab、Docker Compose 等）
- 新增章节或合并章节

## 4. 升级标准

每章升级后应满足以下条件（与 Ch 1 对齐）：

### 4.1 三层深度达标

| 层 | 标准 | 验证方式 |
|----|------|----------|
| 直觉层 | 问题切入用连续场景（电商→数仓→湖仓→RAG→治理），不换场景 | 质量审校清单 Q1-Q3 |
| 机制层 | 每个核心概念按"问题→内部计算路径→代价和失效条件"三段展开，不使用公式化对话开头或功能清单式枚举 | verify-writing-contract + 质量审校清单 Q4-Q6 |
| 工程层 | 有对比表（至少 1 个，含"推荐场景"行和"注意事项"列）+ 故障清单（至少 3 项，含具体症状、根因、缓解措施）+ 选型判断指引 | verify-manuscript-depth + 质量审校清单 Q7-Q10 |

### 4.2 小节规模

- 每章 ≥ 10 小节（与 Ch 1 对齐）
- 每小节 ≥ 800 字
- 总字数 ≥ 12000 字

### 4.3 8-segment 结构保持

升级内容必须嵌入现有 8-segment 结构，不打破已验证通过的骨架。

### 4.4 场景连续性

升级后的内容应沿电商场景演进，每章末尾有显式叙事桥段（区分系统位置和场景位置）：

| 章节 | 系统位置桥段 | 场景位置桥段 |
|------|-------------|-------------|
| Ch 2→3 | SQL 分析是数据库查询能力的上限 | orders 分析遇到大表变慢 |
| Ch 3→4 | 单机边界迫使 OLTP/OLAP 分化 | orders 在 PG 做交易变慢，需要专门的分析系统 |
| Ch 4→5 | OLAP 需要数仓建模 | orders 数据进入数仓需要建模 |
| Ch 5→6 | 数仓需要数据抽取 | orders 从 PG 抽到数仓如何保证一致 |
| Ch 6→7 | 批量历史数据需要批处理 | orders 日/周/月指标需要 Spark/Hive |
| Ch 7→8 | 实时事件需要流处理 | orders 实时流需要 Kafka/Flink |
| Ch 8→9 | 分析查询需要 OLAP 引擎 | 同一 GMV/留存指标在 ClickHouse/Doris 的查询 |
| Ch 9→10 | 非结构化数据需要向量检索 | 企业知识库（产品文档）→ RAG |
| Ch 10→11 | 关系网络需要图查询 | 供应商/客户关系 → 反欺诈/GraphRAG |
| Ch 11→12 | 低成本存储需要湖仓 | orders + 知识数据进湖仓统一管理 |
| Ch 12→13 | 系统复杂需要治理 | 同一平台的 GMV 口径冲突 → 血缘 → 权限 → 审计 |

### 4.5 纵向主线嵌入

每章"小结引出下一章"段末尾加 2-3 句"回溯+推进"桥段，格式：

> **主线回溯**：[上一章在本主线解决了什么问题]。
> **主线推进**：[本章在本主线打开了什么新问题]。
> **未解之问**：[下一章在本主线要解决什么]。

示例（Ch 5→6 建模线）：
> 建模线回溯：Ch 2 用 SQL 聚合做指标，Ch 5 把同样的 orders 数据建模成事实表和维度表。建模线推进：建模解决的是数据如何被组织的问题，但数据如何从业务库到达数仓——抽取、同步、一致性保证——建模本身无法回答。未解之问：数据抽取和同步如何保证与源系统一致？

### 4.6 工程层格式标准

#### 对比表格式（Petrov/DDIA 模式）

| 维度 | 选项 A | 选项 B | 选项 C | 注意事项 |
|------|--------|--------|--------|----------|
| 读性能 | ... | ... | ... | ... |
| 写性能 | ... | ... | ... | ... |
| 一致性 | ... | ... | ... | ... |
| 适用规模 | ... | ... | ... | ... |
| **推荐场景** | 什么场景选 A | 什么场景选 B | 什么场景选 C | -- |

固定列：维度行 + **推荐场景行** + **注意事项列**。每个单元格需包含具体场景触发条件，不能只写功能标签。

#### 故障清单格式（SRE/FMEA 模式）

| 分类 | 故障模式 | 具体症状 | 检测方法 | 缓解措施 | 严重度 |
|------|----------|----------|----------|----------|--------|
| 数据质量 | ... | 必须写具体症状，不能写抽象类别 | ... | 不能只写"用 X 工具" | 高/中/低 |

**具体性规则**：每个故障清单项必须包含：
1. 具体症状（如"Spark job dwd_fact_orders 耗时 45 分钟而非预期 8 分钟"），不能只写抽象类别（如"数据瓶颈：大部分时间在读取数据"）
2. 根因引用本章特定机制（如"city_id 数据倾斜，上海订单量 40x 于平均"）
3. 缓解措施不是"用 X 工具"，而是具体操作（如"对 city_id 做 salting 预处理，shuffle 前先聚合局部结果"）

### 4.7 质量审校清单

每章升级后需通过以下 10 条 yes/no 审校（可部分自动化）：

| # | 审校项 | yes 标准 | 自动化可能 |
|---|--------|----------|-----------|
| Q1 | 问题切入是否使用本章指定连续场景？ | 场景关键词出现 ≥ 3 次 | 部分可 regex |
| Q2 | 核心判断是否明确说明"解决什么/不解决什么"？ | 有"不解决"表述 | verify-writing-contract 已覆盖 |
| Q3 | 场景案例是否引用本章指定电商/数仓/RAG 子场景？ | 子场景关键词出现 | 部分可 regex |
| Q4 | 机制层是否按"问题→路径→边界"展开，而非功能清单枚举？ | 无 yaml 对话开头、无纯功能列表段 | 可检测对话开头标记 |
| Q5 | 每个核心概念是否说明代价和失效条件？ | 有代价或失效条件表述 | verify-writing-contract 已覆盖 |
| Q6 | 机制层是否使用连续场景说明计算路径？ | 场景在 ≥ 3/10 小节中引用 | 部分可 regex |
| Q7 | 工程层对比表是否有"推荐场景"行和"注意事项"列？ | 行列存在 | 可 regex |
| Q8 | 工程层对比表单元格是否包含具体场景触发条件？ | 至少 1 个单元格有具体触发条件 | 需人工 |
| Q9 | 故障清单每项是否有具体症状+根因+缓解措施？ | 至少 3 项满足三要素 | 需人工 |
| Q10 | "小结引出下一章"是否有 5 条主线回溯+推进桥段？ | 桥段存在且格式正确 | 可 regex |

升级完成后需将 `verify-manuscript-depth.mjs` 的最低字符阈值从 4500 提升到 ~12000（与 Ch 1 的 25k chars 对齐，考虑 section-complete 合并后的总量）。

## 8. 约束与假设

### 约束

- 不打破 8-segment 骨架（已验证通过）
- 不重写 Ch 0 和 Ch 1
- 工程层用对比表+故障清单表达，不要求真实平台验证数据
- 保持演进叙事主线不动
- 分 3 批执行，每批完成后验证再进下一批
- 纵向主线嵌入章节（2-3 句桥段），不独立成附录

### 假设

- A1（修正）：Ch 4-9 的 section-complete 机制层质量不均——Ch 7/9 需要机制层重点审校甚至部分重写，其余章节需逐 section 评估后决定是审校还是补写
- A2: Ch 10-13 的机制层补写可以和工程层补写同步进行
- A3（修正）：纵向主线与章节内容不冗余——章节嵌入 2-3 句桥段，主线追踪矩阵确保每章每线新增维度
- A4: 电商场景可以自然延伸到数仓/RAG/治理场景，但需要显式叙事桥段连接
- A5: 每章升级需要 2-3 轮审校-返工周期，不是一次生成通过

### 风险

- R1: AI 生成工程层内容容易泛化——用固定格式模板（对比表固定列+故障清单固定格式）+ 具体性规则约束
- R2: 场景连续性修复可能需要改写已验证内容——增量修改而非全量重写，保持 8-segment 骨架
- R3: Ch 7/9 的公式化机制内容需要部分重写，工作量比纯工程层增量更大
- R4: 每章 2-3 轮审校返工周期，总时间比预估更长
