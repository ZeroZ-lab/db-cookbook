##### 一、什么是事实表
- 定义：记录业务过程中的事实
- 特点：数据量大、不断增长、主要用于分析
- 与维度表的区别
- 典型例子：订单事实表、支付事实表、浏览事实表

##### 二、事实表的粒度（一行代表什么）
- **粒度的重要性**：粒度决定了表的用途和设计
- **常见粒度**：
  - 交易粒度：一行一笔交易
  - 每日粒度：一天一个聚合
  - 用户粒度：一个用户的汇总
- **如何确定粒度**：
  - 业务需求
  - 分析需求
  - 数据来源
- **粒度不一致的后果**：
  - 指标计算错误
  - JOIN时数据爆炸
  - 难以理解和维护

##### 三、事实类型（三种事实表）
- **事务事实表（Transaction Fact Table）**：
  - 定义：记录业务过程中的单个事件
  - 特点：时间点、原子性
  - 例子：订单、支付、点击
  - 适用场景：最常见的事实表类型

- **周期快照事实表（Periodic Snapshot Fact Table）**：
  - 定义：按固定时间间隔记录状态
  - 特点：定期拍照、累积状态
  - 例子：库存日报、账户余额
  - 适用场景：需要跟踪状态变化

- **累积快照事实表（Acculating Snapshot Fact Table）**：
  - 定义：记录业务过程的完整生命周期
  - 特点：多个时间字段、状态更新
  - 例子：订单的生命周期（创建→支付→发货→完成）
  - 适用场景：需要分析流程时长

##### 四、事实表的字段设计
- **事实（度量）**：
  - 定义：可汇总的数值
  - 类型：可加、半可加、不可加
  - 例子：金额、数量、时长
  - 设计原则：所有事实应该是同一粒度

- **外键（指向维度）**：
  - 定义：关联维度表
  - 例子：user_id、product_id、date_id
  - 设计原则：外键必须能唯一关联到维度

- **退化维度（Degenerate Dimension）**：
  - 定义：没有独立维度表的维度
  - 例子：订单号、交易号
  - 使用场景：不需要分析的标识符

- **日期时间字段**：
  - 业务日期
  - 创建时间
  - 更新时间
  - 设计原则：明确时间口径

##### 五、事实表的设计原则
- **原则1：一个事实表只有一个粒度**
  - 为什么：保证所有事实和维度一致
  - 违反的后果：数据混乱

- **原则2：事实应该是可加的**
  - 为什么：便于聚合分析
  - 半可加如何处理：空间维度、库存

- **原则3：外键必须非空**
  - 为什么：保证数据完整性
  - 允许为空的情况：某些维度可选

- **原则4：避免在事实表中存储描述性字段**
  - 为什么：保持事实表简洁
  - 描述性信息放维度表

##### 六、实战案例：订单事实表设计
- **业务场景**：电商平台需要分析订单
- **粒度确定**：一行一个订单（不是订单明细）
- **事实设计**：
  - order_amount：订单金额
  - discount_amount：优惠金额
  - final_amount：实付金额
  - item_count：商品数量
- **维度关联**：
  - user_id → 用户维度
  - date_id → 日期维度
  - channel_id → 渠道维度
- **时间字段**：
  - created_at：创建时间
  - paid_at：支付时间
  - shipped_at：发货时间
- **退化维度**：
  - order_id：订单号

##### 七、常见误区（5个）
- **误区一：事实表粒度不清晰**
  - 问题：一行到底代表什么
  - 后果：JOIN后数据爆炸或丢失
  - 正确做法：明确定义粒度并保持一致

- **误区二：所有字段都放在事实表**
  - 问题：事实表变成宽表
  - 后果：维护困难、难以复用
  - 正确做法：描述性信息放维度表

- **误区三：外键可以为空**
  - 问题：数据关联不完整
  - 后果：分析时数据丢失
  - 正确做法：设计时确保外键非空

- **误区四：不区分事实类型**
  - 问题：用事务事实表表达快照
  - 后果：数据不准确、难以查询
  - 正确做法：根据业务选择合适的事实表类型

- **误区五：事实可以随意加总**
  - 问题：不理解事实的可加性
  - 后果：计算结果错误
  - 正确做法：明确哪些事实可加、半可加、不可加

##### 完整业务场景：从业务库到数仓建模

##### 第二步：设计数仓分层
```
ODS层：
  - ods_users：从users同步
  - ods_products：从products同步
  - ods_orders：从orders同步
  - ods_payments：从payments同步
  - ods_events：从events同步

DWD层：
  - dwd_users：用户清洗、标准化
  - dwd_products：商品清洗、标准化
  - dwd_orders：订单清洗、标准化
  - dwd_order_items：订单明细清洗
  - dwd_payments：支付清洗、标准化
  - dwd_events：事件清洗、标准化

DWS层：
  - dws_users_daily：用户日汇总
  - dws_products_daily：商品日汇总
  - dws_orders_daily：订单日汇总
  - dds_channels_daily：渠道日汇总

ADS层：
  - ads_gmv_daily：GMV日报
  - ads_product_ranking：商品销量排行
  - ads_user_retention：用户留存
  - ads_funnel_analysis：转化漏斗
```

##### 第三步：设计维度模型
- **选择星型模型**
  - 原因：简单、易理解、查询快
  - 适合电商分析场景

- **设计事实表**：
  - fact_orders：订单事实表
  - fact_payments：支付事实表
  - fact_events：行为事实表

- **设计维度表**：
  - dim_users：用户维度
  - dim_products：商品维度
  - dim_channels：渠道维度
  - dim_dates：日期维度

##### 第四步：设计指标体系
- **GMV指标**：
  - 业务定义：已支付订单的金额总和
  - 计算逻辑：SUM(final_amount)
  - 过滤条件：order_status = 'paid'
  - 时间口径：按支付时间
  - 退款处理：扣除退款金额

- **DAU指标**：
  - 业务定义：当天去重登录用户数
  - 计算逻辑：COUNT(DISTINCT user_id)
  - 时间口径：按日期
  - 事件定义：app_open或login

- **留存率指标**：
  - 业务定义：新用户在注册后N日回访的比例
  - 计算逻辑：回访用户数 / 新用户数
  - 新用户定义：某日首次注册
  - 回访定义：N日内发生app_open事件

##### 第五步：从ODS到ADS的完整链路
以GMV日报为例：

**ODS层**：
```sql
ods_orders
  - 直接同步PostgreSQL的orders表
  - 保持原始结构
```

**DWD层**：
```sql
dwd_orders
  - 清洗：过滤无效订单
  - 标准化：状态字段统一
  - 补充：计算字段（如日期）
  - 规范：字段命名
```

**DWS层**：
```sql
dws_orders_daily
  - 按日期聚合
  - 预计算常用指标
  - 多用户复用
```

**ADS层**：
```sql
ads_gmv_daily
  - 面向报表
  - 最终指标口径
  - 业务友好字段名
```

##### 第六步：迭代和优化
- **版本1**：基础分层，简单指标
- **版本2**：增加维度，丰富指标
- **版本3**：优化性能，增加物化
- **版本4**：增加治理，口径统一
