# ClickHouse 表设计说明

## 明细表

`analytics.order_items_wide` 是从 PostgreSQL 订单、订单明细、用户、商品和支付信息展开后的分析宽表。它不承担订单状态约束、库存扣减、支付一致性等 OLTP 职责，只承担面向扫描、聚合和报表的读取职责。

## 分区键

明细表使用 `PARTITION BY toYYYYMM(created_at)`，原因是订单分析通常按自然月归档、重算和清理。月分区能让历史分区删除、冷热分层和月度重跑更直接，同时避免日分区过多造成元数据压力。

汇总表使用 `PARTITION BY toYYYYMM(stat_date)`，与指标观察粒度对齐，便于按月修复指标口径和回填。

## 排序键

明细表使用 `ORDER BY (created_at, category, user_id, order_id)`：

- `created_at` 放在最前，匹配最常见的时间范围过滤。
- `category` 支撑品类维度聚合。
- `user_id` 支撑用户维度下钻。
- `order_id` 保留明细追溯路径，便于从汇总异常回到订单粒度。

这个排序键不适合所有查询。如果主要查询变成“按商品检索最近订单”或“按支付方式聚合”，需要重新评估排序键或增加投影、物化视图、二级跳数索引。

## 汇总表

`analytics.daily_gmv` 使用 `ReplacingMergeTree(updated_at)`，用于承接同一统计日、品类和渠道的重复写入。它解决的是重跑覆盖和幂等写入问题，不解决业务侧重复支付、订单状态回滚和跨系统事务一致性。

## 保留周期

建议把宽表和汇总表分开设置保留策略：

- 明细宽表保留 12-24 个月，满足明细追溯和模型训练取数。
- 汇总表长期保留，用于趋势分析。
- PostgreSQL 仍保留业务真实状态，ClickHouse 删除历史分区不代表业务数据删除。
